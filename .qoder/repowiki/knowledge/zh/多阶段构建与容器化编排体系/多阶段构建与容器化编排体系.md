## 构建系统概览

本项目采用 Maven + Docker 多阶段构建 + GitHub Actions CI/CD 的混合构建体系，以根级 pom.xml 为后端依赖契约中心，以 docker-compose.yml 为本地开发编排枢纽，以 .github/workflows/ci-cd.yml 为自动化流水线入口。

### 1. 后端构建（Maven）
- 父 POM 继承：基于 spring-boot-starter-parent:3.4.13，统一 Spring Boot 生态版本。
- 版本管理策略：通过 properties 集中声明 Java 21、Spring AI 1.1.8、JJWT 0.12.7、Resilience4j 2.3.0 等关键依赖版本；通过 dependencyManagement 引入 spring-ai-bom 统一管理 AI 相关子模块版本。
- 编译配置：启用 -parameters 保留方法参数名（供 Security/JWT 反射使用），Lombok 注解处理器路径显式声明，打包时排除 Lombok。
- 产物命名：finalName=cyber-ai-platform，生成可执行 JAR。

### 2. 前端构建（Vite）
- 独立位于 frontend/ 目录，使用 Vite 8.x 作为构建工具，Node.js 20 环境。
- 构建产物输出至 frontend/dist/，由 CI 上传为 Artifact，也可被后端静态资源目录复用。

### 3. 容器镜像构建（Docker 多阶段）
- 构建阶段：maven:3.9.9-eclipse-temurin-21，先 mvn dependency:go-offline 预拉取依赖以利用 Docker 层缓存，再 mvn clean package -DskipTests 跳过测试加速构建。
- 运行阶段：eclipse-temurin:21-jre 最小运行时镜像，暴露 8080 端口，通过 JAVA_OPTS 环境变量注入 JVM 堆参数。
- 注意：CI 中引用的是 docker/app/Dockerfile，而根目录另有 Dockerfile，两者需保持一致。

### 4. 本地服务编排（Docker Compose）
docker-compose.yml 定义完整微服务栈：
- postgres-pgvector：PostgreSQL 16 + PgVector 向量扩展，含健康检查。
- elasticsearch + kibana：Elasticsearch 8.13.0 及可视化面板。
- logstash：挂载 docker/logstash/pipeline 和 config 目录，只读采集 ./logs。
- ollama：本地大模型推理服务，持久化模型数据。
- app：Spring Boot 应用，通过环境变量注入数据库、AI、JWT、Tavily API Key 等配置，依赖上述服务就绪后启动。
所有服务共享 cyber-network 网络，数据卷 postgres_data、es_data、ollama_data 实现持久化。

### 5. CI/CD 流水线（GitHub Actions）
三阶段并行 Job：
- backend-build-test：JDK 21 + Maven cache → mvn clean verify → 上传 Surefire 报告 → 构建镜像并执行 Trivy 安全扫描（仅 CRITICAL/HIGH）。
- frontend-build：Node.js 20 + npm cache → npm ci → npm run build → 上传 frontend/dist/ 工件。
- code-quality：预留 CheckStyle/Spotless 集成位（当前占位）。
触发条件：push/PR 到 master 或 main 分支。

### 6. 关键约定与约束
- JDK 版本：强制 21（pom.xml / CI / Dockerfile 三者一致）
- 依赖版本：全部在根 pom.xml properties 中声明，禁止子模块硬编码
- 测试数据库：单元测试用 H2 内存库，集成测试用 Testcontainers MySQL
- 日志格式：Logstash JSON 编码器，便于 ELK 管道解析
- 镜像标签：CI 使用 github.sha 作为镜像 tag，支持回滚
- 安全扫描：Trivy 阻断 CRITICAL/HIGH 漏洞
- 前端产物：构建后放入 src/main/resources/static/ 实现前后端单体部署

### 7. 待改进点
- 根目录 Dockerfile 与 docker/app/Dockerfile 存在两份镜像构建脚本，应统一引用避免分歧。
- CI 未对前端构建产物进行质量检查（如 ESLint/Prettier）。
- 缺少发布流程（版本号自动递增、镜像推送至仓库、Helm Chart 更新等）。
